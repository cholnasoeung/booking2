Abstract

This research project presents the design, development, and implementation of an Online Bus Ticket Booking System — a full-stack web application that enables passengers to search for available bus routes, select seats interactively, pay online, and manage their bookings through a personal dashboard. The system also provides a comprehensive administration panel for managing routes, buses, drivers, fleet, employees, and financial reporting.

The system was built using Next.js 16 (App Router) on the front-end and back-end, MongoDB with Mongoose as the database, and NextAuth.js for authentication. Payment processing is handled through Stripe and ABA PayWay, while SMS notifications are delivered via the Twilio API. The system supports twenty-four MongoDB collections covering booking operations, fleet management, human resources, loyalty programs, and analytics.

Testing was carried out through functional, usability, and security testing procedures, confirming that all core features operate correctly. The system demonstrates that a modern, scalable bus booking platform can be developed using open-source technologies and cloud services.

Keywords: Bus Ticket Booking, Next.js, MongoDB, Web Application, Seat Selection, Payment Gateway, Fleet Management, Real-time Notification, Admin Panel

Chapter I

Introduction

The transportation industry plays a crucial role in the economic development of any country. In Cambodia, inter-provincial bus travel remains one of the most common and affordable modes of long-distance transportation. However, the majority of bus ticketing operations continue to rely on traditional, manual methods — physical ticket counters, paper receipts, and verbal communication of seat availability — which are inefficient, error-prone, and inconvenient for modern travelers.

The rapid proliferation of internet access and smartphone adoption in Cambodia has created a significant opportunity to digitize bus ticketing services. Online booking systems eliminate the need for passengers to travel to physical counters, allow real-time seat availability checks, and enable secure digital payments. Bus operators, in turn, benefit from automated booking management, data-driven analytics, and reduced administrative workload.

This research project presents the design and implementation of an Online Bus Ticket Booking System — a comprehensive, full-stack web application that addresses the limitations of traditional bus ticketing. The system provides passengers with a seamless experience from route search and seat selection to payment and ticket download, while equipping administrators with powerful tools to manage the entire bus operation.

1.1 Statement of the Problem

The current manual bus ticketing system in Cambodia presents several challenges for both passengers and bus operators:

  • Inefficiency and inconvenience for passengers: Travelers must physically visit ticket counters or telephone agents to check seat availability and make bookings, often resulting in wasted time and uncertainty about seat confirmation.
  • No real-time seat visibility: Passengers cannot see which seats are available, blocked, or already booked before arriving at the terminal, leading to disappointment and last-minute changes in travel plans.
  • Limited payment options: Manual systems only accept cash, excluding passengers who prefer digital payment methods such as credit cards or mobile banking.
  • Poor record management: Paper-based records are susceptible to loss, damage, and human error. Generating reports, tracking revenue, and monitoring fleet performance require significant manual effort.
  • Absence of driver and fleet management: Bus operators lack a centralized system to track vehicle maintenance, driver schedules, fuel consumption, and incidents.
  • No communication channel for updates: When delays, cancellations, or platform changes occur, there is no reliable mechanism to inform passengers promptly.

These problems indicate a clear need for a digital, integrated bus booking and management system that can serve both passengers and administrators effectively.

1.2 Objectives of the Study

The primary objectives of this research project are as follows:

  1. To design and implement a web-based bus ticket booking system that allows passengers to search routes, view seat availability, and book tickets online.
  2. To develop an interactive seat selection interface that clearly distinguishes between available, booked, blocked, and tier-priced seats.
  3. To integrate secure online payment gateways (Stripe and ABA PayWay) to support both international and local payment methods.
  4. To build a multi-channel notification system that delivers booking confirmations, trip updates, and announcements via email, in-app notifications, and SMS through Twilio.
  5. To implement a comprehensive admin panel covering route management, bus scheduling, fleet and driver management, employee payroll, analytics, and system settings.
  6. To develop a loyalty and rewards program that incentivizes repeat passengers with tiered point-based discounts.
  7. To ensure system security through proper authentication, role-based access control, input validation, and payment signature verification.
  8. To evaluate the system through comprehensive functional, security, and performance testing.

1.3 Significance of the Study

This research project makes several contributions to both the academic and practical domains of information technology:

For Passengers: The system provides a convenient, 24/7 accessible platform for booking bus tickets from any device. Real-time seat selection, multiple payment options, and instant booking confirmation improve the overall travel experience. Passengers can also download PDF tickets, track their booking history, and receive timely notifications about their trips.

For Bus Operators: The admin panel provides a single dashboard for managing all aspects of bus operations — from scheduling routes and buses to tracking driver performance, fuel consumption, vehicle maintenance, and generating financial reports. This significantly reduces manual work and improves operational efficiency.

For Researchers and Developers: This project demonstrates the practical application of modern web technologies — including Next.js App Router, MongoDB, NextAuth.js, Stripe, and Twilio — in solving a real-world logistics problem. It serves as a reference implementation for full-stack web development projects in the context of the Cambodian transportation sector.

1.4 Scope and Limitations of the Study

Scope: The Online Bus Ticket Booking System covers the following functional areas:

  • Passenger-facing features: user registration and authentication, route search, seat selection, passenger details entry, online payment, booking confirmation, ticket download (PDF), booking cancellation with refund calculation, waitlist management, loyalty points, and support chat.
  • Admin-facing features: route and bus management, driver and fleet management, employee and payroll management, booking and user management, check-in system, ratings moderation, lost-and-found management, promo code management, analytics and financial reporting, multi-channel announcements, audit logs, and system settings.
  • The system is designed primarily for the Cambodian inter-provincial bus market and supports both USD and KHR currency display.

Limitations:

  • The system is web-based only; a dedicated mobile application (iOS/Android) has not been developed in this version.
  • Real-time GPS bus tracking and live ETA updates are not included in the current implementation.
  • The Twilio SMS integration requires a paid Twilio account; in environments where Twilio is not configured, SMS notifications are silently skipped.
  • The system has been tested in a development environment; large-scale load testing in a production environment with thousands of concurrent users is outside the scope of this project.
  • Integration with third-party travel aggregator platforms is not covered.

  • Payment Gateway (Stripe / ABA PayWay) — receives payment requests from the system; sends payment confirmation callbacks (webhooks).

Level 1 DFD — Main Processes

Figure 2.2 — Level 1 DFD (Main Processes)

At Level 1, the system is decomposed into six main processes:

  1. Process 1.0 — User Management: Handles registration, email verification, login, session management, profile update, and password reset. Reads from and writes to the User data store.
  2. Process 2.0 — Bus Search & Availability: Receives search parameters from the passenger, queries the Route and Bus data stores, and returns a list of matching departures with seat availability counts.
  3. Process 3.0 — Booking & Payment: Manages seat selection, passenger detail entry, promo code validation, pending booking creation (with TTL), and final booking confirmation after payment callback. Interacts with Booking, PendingBooking, Bus, and PromoCode data stores.
  4. Process 4.0 — Notification Dispatch: Triggered by booking confirmation, cancellation, or admin announcement. Sends emails via Resend API, creates in-app notification records in the Notification data store, and optionally sends SMS via Twilio API.
  5. Process 5.0 — Admin Operations: Receives administrative inputs for managing routes, buses, drivers, fleet, employees, and settings. Reads from and writes to all corresponding data stores.
  6. Process 6.0 — Analytics & Reporting: Aggregates data from the Booking, Bus, and Payroll data stores to produce revenue summaries, occupancy reports, and driver performance metrics.

2.6 System Architecture Design

The Online Bus Ticket Booking System follows a three-tier architecture consisting of the Presentation Layer, Application Layer, and Data Layer. The design principle adopted is Separation of Concerns, where each layer handles a distinct responsibility.

  • Presentation Layer: Built with Next.js 16 App Router using React Server Components for server-rendered pages and Client Components for interactive UI elements. Tailwind CSS and shadcn/ui provide a consistent, responsive design system. Pages include the home/search page, seat selection, passenger details, booking confirmation, user dashboard, and the admin panel.
  • Application Layer: Next.js API Routes and Server Actions handle all business logic. API routes are organized under /app/api/ and cover authentication (/api/auth), bookings (/api/bookings), payments (/api/payments), notifications (/api/notifications), and all admin operations. NextAuth.js manages session creation and validation using JWT tokens.
  • Data Layer: MongoDB is used as the primary database, accessed through Mongoose ODM. A connection pooling utility in lib/mongodb.ts ensures that database connections are reused across serverless function invocations. Twenty-four Mongoose models are defined in the /models/ directory.

External services are integrated at the Application Layer: Resend for transactional email, Stripe and ABA PayWay for payment processing, and Twilio for SMS delivery. All external API credentials are stored in environment variables and accessed via the Settings collection in the database.

3.3 System Modules Implementation

Authentication Module

Authentication is implemented using NextAuth.js v4 configured in lib/auth.ts. The credentials provider validates the user's email and bcrypt-hashed password. Upon successful login, a JWT session token is issued containing the user's ID, name, email, and role. The requireAdmin() and getCurrentUser() utility functions in lib/auth.ts are used by server components and API routes to enforce authentication and role-based access.

Email verification is implemented using a randomly generated token stored in the User document. When a user registers, the system sends a verification email via Resend containing a link with the token. When the link is clicked, the /api/auth/verify-email route validates the token and sets isEmailVerified = true.

Booking and Payment Module

The booking flow follows a pattern designed to prevent seat conflicts under concurrent access. When a user proceeds to checkout, the system creates a PendingBooking document with a 30-minute MongoDB TTL index. This record temporarily holds the selected seats by being referenced during seat availability checks, preventing other users from booking the same seats while payment is in progress.

For Stripe payments, a Checkout Session is created via the Stripe API and the user is redirected to Stripe's hosted payment page. Upon successful payment, Stripe sends a checkout.session.completed webhook to /api/payments/webhook/stripe. The system verifies the webhook signature using the Stripe secret key, creates the confirmed Booking record, locks the seats in the Bus document using MongoDB's $addToSet operator, and creates an in-app notification.

For ABA PayWay, the system generates an HMAC-SHA512 hash of the transaction parameters using the ABA merchant secret key and auto-submits a form POST to ABA's checkout endpoint. ABA sends a callback to /api/payments/callback/aba, where the system verifies the HMAC signature before creating the booking.

Notification Module

The notification system operates on three channels simultaneously:

  1. Email — sent via the Resend API using HTML email templates. Emails are sent for booking confirmation, booking cancellation, password reset, email verification, and admin announcements.
  2. In-App Notifications — stored as documents in the Notification MongoDB collection with a 90-day TTL. The NotificationBell client component polls /api/notifications every 30 seconds and displays a red badge with the unread count.
  3. SMS via Twilio — sent using plain HTTP fetch to the Twilio REST API with Basic authentication. SMS is sent only when Twilio is enabled in Settings and the passenger has a registered phone number.

Admin Panel Modules

The admin panel is a single-page application implemented as a multi-tab interface in app/admin/page.tsx. Each tab corresponds to a distinct management module: Routes, Buses, Bookings, Users, Drivers, Fleet (Bus Details), Maintenance, Fuel Logs, Incidents, Employees, Payroll, Analytics, Finance, Promo Codes, Ratings, Support Inbox, Lost & Found, Audit Logs, Security, and System Settings.

The Add New Bus wizard (app/admin/buses/new/) guides administrators through a 6-step form: (1) Route & Schedule, (2) Vehicle & Driver, (3) Pricing, (4) Amenities, (5) Boarding Stops, and (6) Seat Layout. The seat layout editor allows administrators to configure the seat map, assign tier pricing (Standard / Business / VIP), and block specific seats.

3.4 Database Implementation

The MongoDB database is organized into twenty-four collections. Key implementation decisions include:

  • TTL Indexes: Applied on PendingBooking.expiresAt (30-minute expiry), Notification.createdAt (90-day expiry), WaitingList.expiresAt (7-day expiry), and AuditLog.timestamp (1-year expiry) to ensure automatic document cleanup without manual intervention.
  • Compound Indexes: Applied on the Bus collection for (routeId, date, departureStatus) to optimize the search query. The Booking collection is indexed on (user, status) and (bus, status) for efficient dashboard and admin queries.
  • Unique Constraints: Enforced on User.email, PromoCode.code, LostFound.refNumber, and Rating (user + booking) to prevent duplicate records.
  • Connection Pooling: The lib/mongodb.ts utility caches the Mongoose connection across serverless function invocations using a global variable, preventing connection pool exhaustion in Next.js's serverless deployment environment.

3.5 User Interface Implementation

The user interface is built with Tailwind CSS for utility-first styling and shadcn/ui for accessible, reusable components (Button, Input, Select, Checkbox, Dialog). The design system uses a dark navy color palette (#0c1120 sidebar, #1e3a72 header) consistent with a professional transportation brand.

Key UI screens include:

  • Homepage Search: A centered search form with From City, To City, Date, and Passengers fields, built as a React Server Component with client-side form submission.
  • Seat Selection: An interactive grid component where each seat button reflects its state (available / booked / blocked / business / VIP) through color coding. Clicking a seat toggles its selected state and updates the price summary in real time.
  • Admin Sidebar: A collapsible navigation sidebar with searchable menu items, section grouping with color-coded dots, and a user profile card at the bottom showing the admin's name, email, role badge, and quick-access buttons for Profile and Sign Out.
  • Booking Confirmation: A summary page showing route details, seat numbers, passenger names, total price, payment method, and a QR code for check-in, with options to download the PDF ticket or cancel the booking.

4.1 Testing Procedure

The following testing methodologies were applied:

  • Functional Testing (Black-Box): Test cases were written for each functional requirement, using test inputs to verify that the system produces the expected output. Tested through the browser UI and directly via Postman API calls.
  • Integration Testing: Verified that the payment webhook flow (Stripe CLI forwarding → webhook handler → booking creation → notification dispatch) works end-to-end without failure.
  • Security Testing: Verified that unauthenticated users cannot access protected routes, that session tokens are correctly validated, that payment webhook signatures are verified before processing, and that admin routes reject non-admin sessions.
  • Usability Testing: The system was tested on Google Chrome, Firefox, and Safari browsers on desktop (1920×1080), tablet (768px), and mobile (375px) screen sizes to verify responsiveness.
  • Performance Testing: TypeScript compilation (npx tsc --noEmit) confirmed type safety across all modules. API response times were measured using Postman for key endpoints.

5.1 Future Application

The current implementation of the Online Bus Ticket Booking System establishes a solid foundation for further development. The following enhancements are proposed for future versions of the system:

  • Mobile Application: Developing dedicated iOS and Android applications using React Native would significantly expand the system's reach to mobile users who prefer native app experiences. The existing REST API and NextAuth session infrastructure are already designed to support mobile clients.
  • Real-Time GPS Tracking: Integrating real-time bus location tracking via GPS devices and WebSocket connections would allow passengers to track their bus on a live map. This feature would improve passenger confidence and reduce inquiry calls to operators.
  • AI-Powered Route Recommendations: Machine learning models could analyze historical booking patterns, seasonal demand, and user preferences to recommend optimal routes and departure times to passengers, increasing booking conversions.
  • Multi-Language Support (Internationalization): Adding Khmer language support using Next.js's built-in internationalization would make the system accessible to a broader Cambodian audience who are more comfortable reading in their native language.
  • Third-Party Travel Aggregator Integration: Connecting the system's API with popular travel aggregator platforms would allow tickets to be sold through additional distribution channels, increasing revenue for bus operators.
  • Advanced Analytics with Machine Learning: Implementing predictive analytics to forecast peak booking periods, optimize seat pricing dynamically (yield management), and predict vehicle maintenance needs based on mileage and usage patterns.
  • Biometric Check-In: Integrating facial recognition or fingerprint-based check-in at terminals would replace the current QR code scan system, providing faster and more secure passenger boarding.
  • Carbon Footprint Tracking: Adding a sustainability feature that calculates and displays the CO₂ savings of choosing bus travel over private car travel would appeal to environmentally conscious passengers.

5.2 Conclusion

This research project successfully designed, developed, and implemented a comprehensive Online Bus Ticket Booking System for the Cambodian inter-provincial bus transportation sector. The system addresses the core inefficiencies of traditional manual ticketing by providing passengers with a fully digital experience — from route discovery and seat selection to secure online payment, booking management, and real-time notifications.

The system was built using modern, industry-standard technologies including Next.js 16 App Router, MongoDB with Mongoose, NextAuth.js, Stripe, ABA PayWay, Resend, and Twilio. The architecture follows a clean three-tier design with clear separation between the presentation layer, application layer, and data layer. Twenty-four MongoDB collections were designed to cover all aspects of bus operations, including fleet management, human resources, loyalty programs, and analytics.

All fifteen functional test cases passed successfully. Security testing confirmed that the system correctly enforces authentication, role-based access control, and payment signature verification. The system demonstrates that a full-featured, production-ready bus booking platform can be built by a single developer using open-source technologies and cloud services within an academic project timeline.

The implementation of this system demonstrates the practical applicability of the knowledge gained during the Master of Information Technology program at Build Bright University. It is the researcher's hope that this work will contribute to the digitization of Cambodia's transportation industry and serve as a useful reference for future researchers and developers working on similar systems.
